# IPC Rust 库项目计划书

## 1. 项目背景

本项目目标是开发一个基于 Rust 的 IPC（Inter-Process Communication）进程间调用库，面向本机多进程场景，提供稳定、低心智负担、可演进的调用能力。消息编码格式统一使用 CBOR，以兼顾结构化表达、编码效率与跨语言扩展性。

项目核心原则：

- 自顶而下：先定义用户可见能力、调用模型、错误语义，再逐层下沉到协议、传输和运行时。
- 由外向内：先从 API、示例、集成方式和观测性设计，再收敛到内部模块边界。
- 不脱离用户：每个阶段都以真实使用场景验证，而不是先做“完整内核”再找落地场景。
- MVP 优先：先实现最小可用链路，优先可集成、可调试、可验证，再逐步增强性能和特性。

## 2. 项目目标

### 2.1 总体目标

交付一个可在 Linux / macOS / Windows 主流环境中运行的 Rust IPC 调用库，支持：

- 基于请求/响应模型的进程间调用。
- 统一的消息封装与 CBOR 编解码。
- 超时、错误返回。
- 服务端注册处理器，客户端按方法名发起调用。
- 基础日志、追踪字段和问题定位能力。

### 2.2 非目标

MVP 阶段暂不纳入：

- 分布式跨机器通信。
- 强依赖 IDL 的复杂代码生成体系。
- 流式大文件传输。
- 事务、持久队列、服务治理等重型能力。
- 为所有语言提供完整 SDK。

## 3. 用户与典型场景

### 3.1 目标用户

- Rust 应用开发者，需要拆分本地多进程组件。
- 桌面应用 / Agent / 本地服务开发者，需要稳定的本机 RPC。
- 系统工具开发者，需要替代手写 socket 协议或文本协议。

### 3.2 典型使用场景

1. 主进程调用子进程能力模块，例如索引、渲染、执行器、插件宿主。
2. 本地守护进程对外暴露能力，例如配置查询、任务控制、状态采集。
3. 插件系统中宿主与插件进程隔离，通过统一协议完成调用。

## 4. 需求摘要

### 4.1 功能需求

- 客户端支持连接、发起调用、接收响应、超时控制。
- 服务端支持监听、处理路由、返回结构化错误。
- 支持基础元信息，例如 request_id、method、deadline、trace_id。
- 支持 CBOR 序列化任意结构化消息。
- 支持 Unix Domain Socket 与 Windows Named Pipe 的抽象兼容。

### 4.2 非功能需求

- 易用性：默认 API 简洁，10 分钟内能完成 Hello World。
- 稳定性：异常断连、超时、非法消息必须有明确处理路径。
- 可观测性：至少能看到请求开始、结束、耗时、失败原因。
- 可扩展性：协议字段与传输层实现允许后续演进。
- 性能：MVP 不极限优化，但避免明显低效设计。

## 5. 技术调研方向

项目第一阶段先做调研，不直接进入编码。调研目标不是“找最先进方案”，而是收敛出适合本项目 MVP 的技术组合。

### 5.1 调研主题

1. 传输层方案
   - Linux/macOS：Unix Domain Socket。
   - Windows：Named Pipe。
   - 是否需要统一异步抽象，优先评估 Tokio 生态。
2. 编码方案
   - CBOR 在 Rust 中的主流库能力、性能、兼容性。
   - 重点评估 `ciborium`、`serde_cbor` 等方案。
3. RPC 抽象方案
   - 是否采用显式 `Request/Response` 信封结构。
   - 单连接多路复用是否进入 MVP。
4. 错误模型
   - 系统错误、协议错误、业务错误如何分层。
5. 可观测性
   - `tracing` 与日志字段规范。
6. 跨平台封装
   - 传输接口如何统一，避免上层 API 被平台细节污染。

### 5.2 调研输出物

- 《技术选型记录》：记录候选方案、优缺点、结论。
- 《协议草案》：定义消息结构、错误码、版本字段。
- 《MVP 架构草图》：模块边界、关键数据流、线程/任务模型。
- 《竞品分析简报》：沉淀可借鉴能力与明确不做项。

## 6. 同类产品分析

以下分析以“本机 IPC / 本地 RPC 能力”视角展开，重点看设计取舍，而不是简单做功能罗列。

### 6.1 tarpc

特点：

- Rust 生态中较成熟的 RPC 库。
- 强调类型安全与服务接口定义。
- 更偏通用 RPC，而不是专为本机 IPC 设计。

可借鉴点：

- 清晰的请求/响应抽象。
- 良好的错误边界与异步支持。

不足或不适合本项目的点：

- 接入模型相对更“框架化”。
- 对本机 IPC 场景下的传输封装与最小依赖目标不一定最优。

### 6.2 Cap'n Proto RPC

特点：

- 高性能、零拷贝倾向、带 schema。
- 协议与工具链较完整。

可借鉴点：

- 协议演进与能力模型设计严谨。

不足或不适合本项目的点：

- 心智负担较高。
- 工具链和 schema 体系对 MVP 成本偏大。

### 6.3 gRPC / Tonic

特点：

- 生态成熟、接口定义和工具链完善。
- 更适合跨网络服务调用。

可借鉴点：

- IDL 驱动开发、错误语义、可观测性经验成熟。

不足或不适合本项目的点：

- 对本地 IPC 来说偏重。
- Protobuf + HTTP/2 技术栈会引入超出 MVP 的复杂度。

### 6.4 D-Bus

特点：

- Linux 桌面/系统服务常见 IPC 方案。
- 拥有总线、名称服务、权限体系等机制。

可借鉴点：

- 服务发现、消息类型、系统级 IPC 经验丰富。

不足或不适合本项目的点：

- 平台局限明显。
- 总线模型不适合作为本项目 MVP 的统一抽象。

### 6.5 JSON-RPC over local socket

特点：

- 非正式但常见，易于理解和调试。

可借鉴点：

- 接口模型简单，便于快速落地。

不足或不适合本项目的点：

- JSON 编码冗长，类型表达和效率弱于 CBOR。
- 规范一致性常依赖项目自律。

### 6.6 结论

本项目建议走“轻量本地 RPC”路线：

- 不复制 gRPC 的完整体系。
- 不引入 Cap'n Proto 级别的重工具链。
- 借鉴 tarpc 的抽象清晰度。
- 借鉴 JSON-RPC 的简单调用体验。
- 在本机传输上贴近 Unix Domain Socket / Named Pipe。
- 在消息层使用统一 CBOR 信封协议。

## 7. 产品方案与 MVP 定义

### 7.1 MVP 产品定义

MVP 目标是让用户能在一个工作日内完成以下事情：

1. 定义一个服务端方法，例如 `ping`、`get_status`。
2. 启动服务端监听本地 IPC 地址。
3. 客户端发起调用并拿到结构化响应。
4. 出错时能区分连接失败、超时、服务端业务错误、协议错误。
5. 通过日志定位一次调用链路。

### 7.2 MVP 范围

包含：

- Rust crate 基础结构。
- 客户端/服务端基础 API。
- 请求/响应消息信封。
- CBOR 编解码。
- 单请求单响应模型。
- 超时与取消的最小支持。
- Linux/macOS + Windows 的传输抽象设计。
- 至少 2 个端到端示例。

不包含：

- 代码生成。
- 服务发现。
- 流式调用。
- 多播广播。
- 认证鉴权体系。
- 高级负载均衡或连接池。

### 7.3 推荐的 MVP API 方向

对外优先提供：

- `Server::bind(...)`
- `Server::register(method, handler)`
- `Client::connect(...)`
- `Client::call(method, request)`

先保证简单直接，再考虑：

- typed client stub
- middleware
- 拦截器
- 自动重连策略

### 7.4 MVP 关键约束

为避免阶段 2 返工，MVP 在进入开发前冻结以下约束：

- 采用显式请求/响应信封，不做隐式约定或裸 payload 传输。
- 采用长度前缀分帧，不依赖传输层天然消息边界。
- 默认长连接复用，但不在 MVP 内支持单连接多路复用。
- 默认支持超时，不定义跨进程取消协议。
- `deadline_ms` 固定为 Unix epoch 毫秒时间戳，由客户端基于调用超时推导写入。
- 不内建自动重试，也不做后台保活式自动重连；是否重试由调用方按幂等性自行控制。
- 客户端调用一旦发生本地超时，当前连接必须标记为不可复用；后续新调用开始前，`Client` 可按需惰性重建连接，但不得重试已经超时的那次调用。
- 先交付跨平台统一抽象和端到端链路，再讨论 typed stub 与 middleware。

### 7.5 帧格式与协议边界

MVP 协议采用“固定头 + CBOR 信封”的分层方式：

- 帧格式：`u32_be length` + `payload bytes`。
- `length` 表示后续 payload 字节长度，不包含长度字段本身。
- `payload` 为单个完整 CBOR 文档，对应一个请求帧或响应帧。
- 默认最大帧大小需在协议草案中明确，建议 MVP 先限制在 4 MiB 以内。
- 读取到非法长度、超出上限、CBOR 解码失败时，统一归类为协议错误。

采用长度前缀而不是依赖 Named Pipe 或其他传输语义，可避免平台差异直接泄漏到 `codec` 与 `runtime` 层。

### 7.6 超时、取消与重试语义

MVP 明确采用以下语义：

- `Client::call` 必须支持调用级超时控制，可由参数或 client 默认配置提供。
- 超时行为默认只影响客户端等待，不保证服务端中止处理。
- 请求信封中保留 `deadline_ms` 字段，固定表示 Unix epoch 毫秒时间戳，服务端可用于拒绝已过期请求。
- MVP 不定义单独的 cancel 帧；调用 future 被丢弃不等价于服务端取消执行。
- MVP 不提供自动重试策略，文档中只说明推荐由业务侧对幂等方法显式重试。
- 为避免晚到响应污染后续请求，客户端在本地等待超时后必须关闭当前连接，并将该次调用返回为本地 `Timeout`；后续新调用可触发惰性重连，但不得隐式重试已经超时的请求。
- 服务端若基于 `deadline_ms` 或本地处理期限返回 `timeout`，语义仅表示“服务端停止等待并拒绝继续为该调用承担时限承诺”，不要求强制中止已开始执行的业务逻辑。
- 若未来需要取消协议，应作为后续版本的显式扩展能力，而不是在 MVP 中半实现。

## 8. 架构思路

### 8.1 自顶而下设计顺序

第一层：用户 API

- 用户如何启动服务、注册方法、发起调用、处理错误。

第二层：协议模型

- Request / Response / Error / Metadata 的结构定义。

第三层：运行时模型

- 连接管理、请求派发、超时控制、任务并发。

第四层：传输实现

- Unix Domain Socket / Named Pipe。

第五层：编码与底层优化

- CBOR 编码、缓冲区管理、性能优化。

### 8.2 由外向内设计顺序

先做：

- 使用示例
- API 草图
- 错误枚举
- 日志字段
- 测试用例草图

再做：

- 协议字段定稿
- 内部 trait 抽象
- 传输细节实现

这样可以避免内部实现先行，最后 API 不可用或难以集成。

### 8.3 连接与并发模型

MVP 运行时模型先采取保守设计：

- 一个 `Client` 默认维护一个长连接。
- 同一连接上，MVP 仅支持串行 in-flight request，即前一个请求完成后再发送下一个请求。
- `Client` 对外可保留 `&self` 调用接口，但并发调用必须在库内串行排队，且文档中需明确这一语义。
- 服务端可按连接并发处理多个连接，但单连接内先按串行请求处理。
- 若单次调用在客户端侧超时，该连接必须退出连接池或内部复用集合，不得继续承载后续请求。
- 失效连接在后续新调用开始前可由 `Client` 惰性重建，但该重建只用于后续调用，不得改变已返回的超时结果。
- 若后续引入多路复用，需新增 request correlation 和乱序响应约束，但这不属于 MVP。
- 服务端必须提供最大连接数、最大帧大小、最大处理超时等基础保护参数。

该模型牺牲部分吞吐，但能显著降低协议、状态机和跨平台实现复杂度，适合作为首版基线。

### 8.4 错误模型基线

错误需分为本地错误与线协议错误两层：

- 本地错误：建立连接失败、地址不可达、权限不足、读写失败、超时。
- 协议错误：非法帧长度、版本不兼容、消息类型不合法、CBOR 解码失败、缺少必填字段。
- 业务错误：方法不存在、参数非法、处理器返回业务失败。
- 服务端内部错误：处理器 panic、未分类内部异常、资源耗尽。

MVP 中服务端 handler 的返回通道只承载业务错误或服务端内部错误，不直接复用客户端对外 `Error` 类型；建连失败、I/O 失败、协议解析失败等本地错误由 runtime 产生并在本地 API 暴露或映射。

MVP 建议至少冻结以下线协议错误码：

- `method_not_found`
- `invalid_request`
- `decode_error`
- `timeout`
- `internal_error`

其中传输层 I/O 错误默认不跨线传输，由本地 API 直接暴露。

此外，MVP 需冻结以下映射规则：

- 本地超时、建连失败、连接断开、读写失败，统一作为本地错误暴露，不经线协议回传。
- 可安全解析的非法请求统一返回 `invalid_request`，并在响应后关闭该连接，避免实现保留“半可信连接”状态。
- 不可安全解析的错误，例如超大帧、非法 CBOR、帧不完整，统一直接关闭连接，不返回错误响应。
- 远端显式返回的 `invalid_request` 统一映射为客户端 `Protocol` 错误；`method_not_found`、`decode_error`、`timeout`、`internal_error` 统一映射为客户端 `Remote` 错误。

## 9. 模块划分建议

- `api`：对外暴露的客户端、服务端、高级类型。
- `protocol`：消息信封、错误码、元数据、版本。
- `transport`：跨平台传输抽象与具体实现。
- `codec`：CBOR 编解码与帧格式。
- `runtime`：请求分发、超时、连接状态。
- `observability`：日志、trace 字段、指标预留。
- `examples`：最小示例与参考接入。

### 9.1 模块边界补充

- `transport` 只负责可靠读写字节流与地址绑定，不负责协议理解。
- `codec` 负责分帧、CBOR 编解码、消息大小校验。
- `runtime` 负责请求生命周期、超时控制、处理器调度和响应回写。
- `protocol` 只定义稳定的数据结构、错误码和版本常量。
- `api` 不直接暴露平台特有类型，统一通过库内地址抽象承载。

## 10. 实施阶段计划

### 阶段 0：技术调研与方案收敛

目标：

- 形成技术选型结论与竞品分析。
- 明确 MVP 边界。

产出：

- 技术调研文档
- 协议草案
- 架构草图
- 风险清单
- 分帧方案与最大消息大小结论
- 平台支持矩阵与首版验证范围
- 平台验证资源清单，例如 CI runner、人工验证机器或明确缺口

预计周期：

- 3 到 5 个工作日

### 阶段 1：接口优先设计

目标：

- 固化用户视角 API 和示例代码。
- 冻结进入核心开发前必须拍板的对外语义与内部边界。

产出：

- API 草案
- 示例程序
- 错误模型文档
- 首批测试用例设计
- 连接/并发模型定稿
- 超时与取消语义定稿
- API 与运行时冻结决议记录，固定以下事项：
  - `trace_id` 缺省时由客户端在每次调用前自动生成 UUID v4 字符串，并写入请求信封
  - Windows Named Pipe 地址由 `IpcAddress::NamedPipe(String)` 承载逻辑名称；运行时统一规范化为 `\\.\pipe\<name>`
  - `Server::register` 最终签名固定为 `&mut self`
  - handler 类型擦除边界固定为“原始 CBOR payload bytes -> 编码后响应 bytes / ServiceError”
  - `Client` 本地超时后连接失效，后续新调用允许惰性重连，但不得隐式重试前一次请求
  - 服务端最小停服接口固定为 `serve_until(shutdown)`，`serve()` 仅作为永久运行的便捷包装
  - 默认安全策略固定为：Unix Domain Socket 使用受控目录与最小权限；Windows Named Pipe 默认限制为当前用户可访问

阶段出口条件：

- Hello World 与失败调用示例已按冻结后的 API 重新过一遍，不保留二义性接口写法。
- 协议草案、API 草图与错误模型之间不存在未决字段或未决语义。
- 默认权限策略、地址规范化规则和错误映射规则已经写入文档，而不是只停留在风险说明。
- 若上述七项核心决策有任一未冻结，不进入阶段 2。

预计周期：

- 3 到 5 个工作日

### 阶段 2：MVP 核心开发

目标：

- 在一个主开发平台上打通 client -> transport -> server -> response 全链路。
- 完成跨平台抽象层，并在至少一个非主平台完成真实链路验证。

产出：

- 主开发平台基线实现与测试夹具
- 可运行 MVP 库
- 基本单元测试与集成测试
- 最小可用示例
- 跨平台传输抽象
- 已具备环境的平台真实链路验证结果，以及未覆盖平台的阻塞项记录
- 发布口径草案，明确以下状态：
  - 已完成验证的平台
  - 仅完成编译或抽象适配的平台
  - 尚未验证的平台与阻塞原因

阶段出口条件：

- 主开发平台上已通过协议草案最小测试矩阵中的关键自动化用例：成功调用、方法不存在、非法长度、超大帧、非法 CBOR、缺少必填字段、服务端业务解码失败、客户端超时、`deadline_ms` 过期拒绝、本地超时后新连接继续、`version != 1`、服务端内部错误。
- 至少一个非主平台完成真实链路验证；若未完成，则阶段 2 不达标，M4/M5 不得宣称跨平台 MVP 已达成。
- 已确认客户端超时后连接失效策略、协议错误断连策略和 `deadline_ms` 编码规则在实现与文档中一致。
- 已确认 Unix Domain Socket 默认权限策略和 Windows Named Pipe 默认安全策略在实现与文档中一致。
  - Unix 当前实现：父目录 `0700`，socket 文件 `0600`，并在绑定前清理旧 socket 文件。
  - Windows 当前实现：默认启用 `first_pipe_instance` 保护，并拒绝 remote clients；“仅当前用户可访问”的安全描述符仍作为后续增强项。

预计周期：

- 20 到 25 个工作日

### 阶段 3：稳定性与易用性增强

目标：

- 补齐超时、异常断连、日志、错误表现。
- 完成文档收口、可观测性补强与剩余平台验证。

产出：

- 错误处理完善
- 日志追踪完善
- 文档与使用指南
- 平台支持说明，明确“已验证 / 兼容性目标 / 未验证”分层
- 发布检查清单与剩余风险说明

阶段出口条件：

- 发布说明与验收标准口径一致，不把未验证平台写成已支持平台。
- 使用指南覆盖地址命名、默认权限策略、超时语义和并发排队语义。

预计周期：

- 5 到 8 个工作日

## 11. 里程碑

- M1：完成技术调研与竞品分析，冻结 MVP 范围。
- M2：完成 API 设计评审和协议草案评审。
- M3：主开发平台完成最小调用链路的端到端联通。
- M4：完成跨平台传输抽象，并在至少一个额外已具备验证环境的平台上完成真实链路验证；未验证平台仅计为兼容性目标。
- M5：按“已验证 / 兼容性目标 / 未验证”口径发布 MVP 版本并提供接入示例。

## 12. 风险与应对

### 12.1 跨平台 IPC 差异过大

风险：

- Unix Domain Socket 与 Named Pipe 在语义和实现细节上差异明显。

应对：

- 上层统一抽象只暴露必要能力。
- MVP 避免依赖难统一的高级特性。

### 12.2 过早设计复杂协议

风险：

- 过多为未来扩展设计，导致首版实现迟缓。

应对：

- 协议保留版本和扩展字段，但只实现 MVP 必需语义。

### 12.3 API 不贴近用户

风险：

- 内部实现优先，导致用户接入复杂。

应对：

- 先写示例和伪代码，再反推 API。
- 每阶段都用真实场景回看是否易用。

### 12.4 调试困难

风险：

- IPC 问题往往表现为超时、断连、无响应，排障成本高。

应对：

- 从 MVP 起就定义 request_id、trace_id、错误分类和关键日志。

### 12.5 平台承诺过早过满

风险：

- 若只完成抽象而未在三平台真实验证，计划目标与交付物会发生偏差。

应对：

- 在阶段 0 明确平台支持矩阵。
- 在阶段 2 输出真实平台验证结果，而不是仅说明“理论支持”。

### 12.6 本机安全边界模糊

风险：

- IPC 地址、默认权限和连接主体未约束时，桌面应用或本地守护进程可能暴露给不期望的本机进程。

应对：

- 在阶段 1 冻结地址命名规范与默认权限策略，不等到阶段 3 才补文档。
- Unix Domain Socket 默认使用受控路径和最小权限，Named Pipe 默认应用当前用户可访问的安全描述符策略。

### 12.7 验证资源不足导致计划失真

风险：

- 若没有 macOS / Linux / Windows 的真实验证环境，三平台承诺会停留在设计层，导致里程碑与验收标准名义达成、实际未验证。

应对：

- 在阶段 0 明确可用 runner、测试机和人工验证路径。
- 若某平台在阶段 2 前仍无验证资源，将其从“必须验收项”降为“兼容性目标 + 风险说明”，避免交付口径失真。

### 12.8 API 冻结不彻底导致阶段 2 返工

风险：

- 若 `register` 形态、地址抽象或 handler contract 在开发中继续摇摆，runtime、示例和测试会同时返工。

应对：

- 阶段 1 结束前形成 API 冻结记录，未冻结不进入阶段 2。
- 所有示例代码必须基于冻结后的 API 重新校验一遍，避免文档与实现各自演化。

## 13. 验收标准

满足以下条件即可认为 MVP 达标：

- 用户可在示例中完成一次成功调用与一次失败调用。
- 服务端与客户端错误能被结构化区分。
- 消息格式统一为 CBOR，协议字段文档齐备。
- 分帧格式、最大消息大小、错误码表均有文档定义。
- 核心 API 控制在少量关键入口，易于理解。
- 提供基础测试、示例和使用说明。
- 至少在已具备真实验证环境的平台上完成端到端调用验证；若缺失某平台环境，必须在发布说明中显式标注未验证平台与剩余风险。
- 对协议草案最小测试矩阵中的关键状态机路径均有自动化测试覆盖，至少包括：方法不存在、非法长度、超大帧、非法 CBOR、缺少必填字段、服务端业务解码失败、客户端超时、`deadline_ms` 过期拒绝、本地超时后新连接继续、`version != 1`、服务端内部错误。
- 文档明确客户端并发调用排队语义、`deadline_ms` 编码规则、超时后的连接失效策略和协议错误断连策略。
- 文档明确地址规范化、Unix Domain Socket 默认权限和 Windows Named Pipe 默认安全策略。
- 文档明确区分“已验证支持的平台”和“仅作为兼容性目标的平台”，不得混用“支持”表述。

## 14. 下一步建议

建议立即执行以下动作：

1. 输出技术调研清单与候选库列表。
2. 编写 API 草图和 Hello World 示例伪代码。
3. 定义协议信封初稿，包括 `version`、`request_id`、`method`、`deadline_ms`、`payload`、`error`。
4. 定义帧格式、最大消息大小和最小错误码表。
5. 选定 MVP 运行时与序列化依赖，并冻结连接/并发模型。
6. 补充 API 冻结记录，拍板 `register` 签名、地址抽象、handler contract、超时后惰性重连语义、最小停服接口和默认安全策略。
7. 明确主开发平台、额外验证平台和当前缺失的验证资源。
8. 在方案评审通过且阶段出口条件满足后进入核心链路开发。

## 15. 后续实施计划

本节用于把前述阶段目标进一步拆成可排期、可分配、可验收的工作项。默认前提是：阶段 1 冻结决议已经完成，且协议草案、API 草图、测试矩阵三者口径一致。

### 15.1 执行顺序

建议按以下顺序推进，而不是并行摊开：

1. 先完成阶段 0 收尾，确认依赖选型、主开发平台和额外验证平台。
2. 再完成阶段 1 冻结，把 API、错误语义、地址规范、安全策略和测试矩阵锁死。
3. 阶段 2 先打通主开发平台最小链路，再补协议异常路径，最后做非主平台真实验证。
4. 阶段 3 再补文档、观测性和剩余平台验证，不把这些工作反向塞回阶段 2。

### 15.2 阶段 0 工作分解

工作项：

1. 选定异步运行时与核心依赖，至少覆盖 `tokio`、CBOR 编解码库、错误处理与日志方案。
2. 输出平台支持矩阵，明确主开发平台、额外验证平台、尚未具备环境的平台。
3. 列出验证资源清单，包括本地机器、CI runner、人工验证路径和缺口。
4. 完成竞品与候选方案对比，给出“采用 / 不采用 / 暂缓”的结论。

完成定义：

- 形成技术选型记录。
- 平台验证资源状态明确，不再用“后续补环境”作为阶段 2 默认前提。
- 依赖选择足以支撑阶段 1/2，不保留会阻断实现的重大未决项。

### 15.3 阶段 1 工作分解

工作项：

1. 将 `api-sketch.md` 中的地址抽象、`register` 形态、`call` 语义、停服接口全部同步到示例。
2. 将 `protocol-draft.md` 中的错误映射、断连策略、超时规则同步到 `plan.md` 验收标准。
3. 产出至少两个示例：
   - Hello World 成功调用
   - 失败调用与错误分支处理
4. 把协议草案最小测试矩阵转成测试设计清单，标明单元测试、集成测试、跨平台验证各自覆盖哪些项。
5. 固定默认安全策略，明确 Unix 路径约束和 Windows pipe 安全描述符的默认行为。

完成定义：

- `plan.md`、`protocol-draft.md`、`api-sketch.md` 不再互相引用未决项。
- 示例代码与冻结后的 API 一致，不存在一个文档写 `self`、另一个文档写 `&mut self` 的情况。
- 每条关键协议语义都能在测试设计中找到对应验证方式。

### 15.4 阶段 2 工作分解

建议按 5 个子阶段推进。

#### 15.4.1 子阶段 A：项目骨架与协议类型

工作项：

1. 建立 crate 目录结构，至少包含 `api`、`protocol`、`codec`、`transport`、`runtime`、`observability`、`examples`。
2. 落地请求/响应信封、错误体、版本常量和地址抽象。
3. 补齐协议字段级单元测试，例如 `ok/error` 组合约束、`version` 校验和错误码映射。

完成定义：

- 协议类型可以独立编译和测试。
- 不依赖真实 IPC 传输即可验证基本编码模型。

#### 15.4.2 子阶段 B：codec 与本地状态机

工作项：

1. 实现长度前缀分帧和最大帧大小校验。
2. 实现 CBOR 编解码与原始 payload bytes 的适配。
3. 实现非法长度、超大帧、非法 CBOR、缺少字段、版本不兼容的本地处理路径。
4. 实现客户端超时后连接失效、后续调用惰性重连的状态机。

完成定义：

- 协议草案中的异常分支在不依赖真实平台传输的测试里可复现。
- 连接失效与重连语义已经可测，不再只是文档承诺。

#### 15.4.3 子阶段 C：主开发平台传输落地

工作项：

1. 在主开发平台实现 listener、accept、连接读写和资源清理。
2. 打通 `Client::connect -> call -> Server::register -> serve_until` 的最小成功链路。
3. 实现方法不存在、服务端业务解码失败、服务端内部错误、过期 `deadline_ms` 的响应路径。
4. 落地默认安全策略，包括路径创建、权限设置和清理行为。

完成定义：

- 主开发平台上已可跑通成功与失败调用示例。
- 默认安全策略体现在实现中，而不只是文档描述。

#### 15.4.4 子阶段 D：测试补齐与稳定性基线

工作项：

1. 将最小测试矩阵全部转成自动化测试。
2. 按连接生命周期补齐状态机测试，重点覆盖：
   - 本地超时后旧连接失效
   - 下一次调用在新连接继续
   - 协议错误后的断连行为
3. 为示例和关键测试加基础日志字段，确保排障时能看到 `request_id`、`trace_id`、`method`、`error.code`。

完成定义：

- 阶段 2 出口条件涉及的测试全部自动化。
- 失败路径的日志字段已经足够定位问题。

#### 15.4.5 子阶段 E：非主平台验证

工作项：

1. 完成至少一个非主平台的真实链路验证。
2. 记录验证环境、执行步骤、通过项和阻塞项。
3. 对未验证平台仅保留“兼容性目标”口径，不提前承诺已支持。

完成定义：

- 至少一个非主平台有可复现的真实验证记录。
- 发布说明里的平台状态与实际验证结果一致。

### 15.5 阶段 3 工作分解

工作项：

1. 整理接入文档，覆盖地址命名、权限策略、超时语义、错误分类、并发排队语义。
2. 完善观测性，至少统一日志字段和关键事件名称。
3. 收敛发布检查清单，明确已验证平台、兼容性目标平台、未验证平台。
4. 若具备资源，补剩余平台验证；若不具备，补风险说明与后续计划。

完成定义：

- 用户只看文档即可完成成功调用与失败排障。
- 发布口径、文档口径、测试口径三者一致。

## 16. 交付清单

在进入 MVP 发布评审前，至少应准备以下交付物：

1. Rust crate 源码与可运行示例。
2. `plan.md`、`protocol-draft.md`、`api-sketch.md` 的冻结版文档。

2. 主开发平台和至少一个额外验证平台都已落实到具体环境。
3. 协议草案最小测试矩阵已经映射成测试任务列表。
4. crate 模块边界和责任划分已确定，不再保留结构性摇摆。
5. 客户对平台口径、MVP 边界和非目标范围无异议。
